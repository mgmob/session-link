# Баг-репорт: `session_link` не передаёт тело вопроса связанной сессии

> Статус: НЕ починено (workaround — read-only разбор вручную).
> Автор отчёта: pi-сессия `019f22ae-dd95-7659-9ff4-10d4284a1ca1` (этап 5 docs/14).
> Объект: инструмент `session_link` + механизм handoff-link в pi.

---

## 1. Симптом

При вызове инструмента `session_link` с развёрнутым аргументом `question` —
связанная (предыдущая) pi-сессия получает **только инструкцию-обёртку**, а
**сам текст вопроса не доходит** (поле тела вопроса пустое).

Связанная сессия дважды честно сообщила об этом в ответе:

> «в вашем сообщении нет самого вопроса — есть только инструкция отвечать на
> вопрос "ниже", а ниже пусто. Пожалуйста, продублируйте вопрос…»

> «Снова пришла только инструкция — самого вопроса нет. В сообщении сказано
> "Answer the question below", но ниже вопроса не следует (пусто).»

## 2. Воспроизводимость

- **2 попытки из 2** — стабильно (не разовая сеть/таймаут).
- Полный развёрнутый вопрос (несколько абзацев) оба раза проигнорирован.

## 3. Исторический прецедент (системность бага)

В `.userdocs/SESSIONS.md`, строка сессии `019f14b1` (этап 2 docs/08),
зафиксировано дословно:

> «`session_link` технически неисправен (3 попытки) — решение вариант C
> принято пользователем напрямую»

Тот же симптом: связанная сессия не получает тело вопроса. Значит баг не
этой сессии, а **устойчивая проблема механизма handoff-link** в pi.

## 4. Гипотезы о root cause (для проверки)

### H1. Формат обёртки промпта / место подстановки вопроса
Связанная сессия получает системную/обёрточную инструкцию вида
`...Answer the question below:\n{QUESTION}`, но `{QUESTION}` подставляется в
**отдельное поле** (user-role message), которое либо не вкладывается в ту же
доставку, либо парсится как пустое. Инструкция-обёртка доходит, тело — нет.

### H2. Рассинхрон `askCommand` (массив argv) vs инструмент-параметр `question`
В `handoff.json` поле `askCommand` (стр. 11–22) — массив argv с плейсхолдером
`{QUESTION}` как **последним элементом**. Если pi internally дёргает связанную
сессию по этому массиву через CLI
(`pi --session ... "{QUESTION}"`), а инструмент `session_link` передаёт вопрос
в свой параметр `question` — возможен рассинхрон: инструмент пишет вопрос в
одно место, а resumed-сессия читает из argv-плейсхолдера, который остаётся
литералом `{QUESTION}` или пустой строкой.

### H3. Цепочка `parentHandoffPath`
В текущем handoff (стр. 24) `parentHandoffPath` →
`handoff-2026-07-02T11-21-06-249Z.json` (т.е. указывает на
**предыдущий-предыдущий** handoff, не на прямой parent). Возможно resumed-сессия
читает контекст из этой цепочки, где вопроса нет, а свежий `question` не
пробивается через лишнее звено.

### H4. tool-schema vs runtime-проброс
Инструмент `session_link` принимает `{question, handoffPath?, clarifications?}`.
Возможно runtime корректно прокидывает `question` только когда returns
`kind:"answer"`, а для первоначального запроса путь данных другой.

## 5. Что нужно починить (зоны фикса)

1. Маппинг `question`-аргумента инструмента → фактическое user-message-тело,
   которое видит resumed-сессия (точка потери тела вопроса).
2. Парсинг плейсхолдера `{QUESTION}` в `askCommand` при resume-вызове (если
   resume идёт через CLI-массив).
3. Логирование на стороне resumed-сессии ровно того сообщения, которое она
   получает (длина, первые байты) — для диагностики «пустой user-message vs
   отсутствующий токен».

## 7. Workaround (применён)

`session_link` признан ненадёжным — до фикса разбор неясностей ведётся
read-only инструментами (read/grep/find/git) по репозиторию, без запросов к
предыдущей сессии. В текущей сессии этапа 5 так был решён вопрос про dna4
AGENTS.md (установлено, что в dna3 он создан ручным коммитом `585e236`, а не
из scaffold — `templates/pi` никогда не содержал AGENTS.md).

## 8. Доп. контекст (важно для того, кто чинит)

Handoff-документ правил **другой агент в системе** (новейший Fable 5 от
Anthropic), а не сама pi-сессия `019f2274`. Поэтому в контексте предыдущей
pi-сессии могут быть **не все ответы** на вопросы follow-up — часть намерения
живёт только в head'е Fable-агента, написавшего handoff. Это отдельный риск
достоверности handoff-поля `summary`/`nextStep` vs реальный контекст
pi-сессии (возможные расхождения session_link не выловит, т.к. неисправен).
